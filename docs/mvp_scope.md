# MVP Scope — Diplomacy AI for MODHOI4

## Цель MVP
Собрать минимально жизнеспособную версию дипломатической подсистемы, где:
- игрок может вести диалог с AI-страной;
- AI-страны могут инициировать базовые контакты друг с другом;
- решения учитывают простые социально-экономические параметры и фиксируются в хронике.

## In Scope (входит в MVP)

### 1) Стартовый охват стран
Ограниченный стартовый набор **5–10 стран**. Рекомендуемый первичный набор (7 стран):
1. Германия
2. СССР
3. Великобритания
4. Франция
5. США
6. Италия
7. Япония

> Допустимо заменить конкретные страны, но итоговый набор в MVP должен оставаться в диапазоне 5–10.

### 2) Типы дипломатических запросов (MVP)
Ограничить систему до **3–5 типов запросов**. Базовый набор (5 типов):
1. **Ненападение** (предложение/принятие/отказ).
2. **Торговая сделка** (ресурсы/деньги в простой форме).
3. **Военный доступ** (доступ на территорию).
4. **Гарантия независимости**.
5. **Улучшение отношений** (soft-diplomacy действие).

### 3) Базовые игровые параметры
В модели решения должны участвовать следующие параметры:
- **trust** (доверие),
- **fear** (страх),
- **influence** (влияние),
- **денежный капитал** (money budget / treasury).

### 4) Динамические события
Реализовать минимум **3 динамических события**, которые влияют на trust/fear/influence и/или денежный капитал.

Примеры событий:
- Пограничный инцидент (рост fear, падение trust).
- Выгодное торговое соглашение (рост trust, рост money).
- Политический кризис/смена курса (изменение influence и trust к отдельным блокам стран).

### 5) Хроника и time skip
После выполнения **time skip** должна формироваться/обновляться хроника:
- список ключевых дипломатических действий,
- срабатывания динамических событий,
- изменения статусов отношений (если есть).

## Out of Scope (не входит в MVP)
- Полный охват всех стран/фракций игры.
- Продвинутое многошаговое стратегическое планирование AI.
- Сложные многосторонние договоры и коалиционные переговоры.
- Детальная экономическая симуляция (рынки, инфляция, цепочки поставок и т.д.).
- Полноценная локализация на все языки.
- Полная UI/UX-полировка и финальная визуальная интеграция.
- Многопользовательский режим и сетевые дипломатические сессии.

## Definition of Done (DoD) для MVP
MVP считается завершённым, когда одновременно выполнены все пункты:

1. **Работает диалог игрока с AI-страной**
   - Игрок может инициировать минимум один дипломатический запрос из MVP-набора.
   - AI формирует валидный ответ и состояние обновляется.

2. **Работает AI-to-AI контакт в базовом виде**
   - Минимум один тип дипломатического взаимодействия между двумя AI-странами инициируется без участия игрока.

3. **Есть trust/fear/influence + денежный капитал**
   - Все 4 параметра присутствуют в данных страны.
   - Параметры влияют минимум на одно дипломатическое решение.

4. **Есть минимум 3 динамических события**
   - События автоматически срабатывают по простым условиям/вероятности.
   - События меняют состояние мира (параметры, отношения, деньги и т.д.).

5. **Есть хроника после time skip**
   - После time skip формируется читабельная запись ключевых изменений за период.

## Метрики качества MVP
Для базового мониторинга качества и устойчивости системы отслеживаются:

1. **Доля валидных JSON-ответов**
   - Формула: `valid_json_responses / total_ai_responses`.
   - Цель MVP: **≥ 95%**.

2. **Средняя задержка ответа AI**
   - Метрика: среднее/медиана времени ответа (мс).
   - Цель MVP: зафиксировать baseline и держать в договорённом бюджетe (например, p50/p95).

3. **Частота fallback**
   - Формула: `fallback_count / total_ai_responses`.
   - Цель MVP: минимизировать; целевой ориентир **≤ 10%**.

4. **Число “значимых” событий за игровой год**
   - Метрика: количество событий, изменивших хотя бы один из ключевых параметров (trust/fear/influence/money) выше заданного порога.
   - Цель MVP: обеспечить наблюдаемую динамику мира (например, не менее 10–20 значимых событий/год на весь пул стран).

## Примечания по внедрению
- Порог “значимости” события и бюджет задержки лучше зафиксировать в конфиге.
- Для прозрачности рекомендуется логировать: вход запроса, выход AI (raw), результат валидации JSON, факт fallback, latency.
