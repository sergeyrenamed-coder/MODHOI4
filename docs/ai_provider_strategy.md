# AI Provider Strategy

## Цель
Определить минимальный, но расширяемый контракт AI-провайдера для генерации дипломатических намерений в симуляции и зафиксировать стратегию выбора провайдеров под разные MVP-профили.

## Интерфейс провайдера

```text
generate_diplomatic_intent(context) -> structured_json
```

### Контракт `context` (минимум)
- `actor_tag`: string — инициатор дипломатического действия.
- `target_tag`: string — целевая страна/фракция.
- `world_state`: object — релевантные сигналы мира (напряженность, войны, гарантия независимости, отношения).
- `memory_window`: array — последние значимые дипломатические события.
- `constraints`: object — hard rules движка (запреты, cooldown, доступные action types).
- `profile`: enum — `mvp_low_end | mvp_full_simulation`.

### Контракт `structured_json` (минимум)
- `intent_type`: enum (`improve_relations`, `non_aggression_pact`, `guarantee`, `faction_invite`, `justify_war`, `none`).
- `confidence`: number `[0..1]`.
- `priority`: integer `[0..100]`.
- `rationale`: short string (1–2 предложения для отладки/телеметрии).
- `valid_for_days`: integer.
- `meta`:
  - `provider`: string.
  - `model`: string.
  - `latency_ms`: integer.

### Обязательные инварианты
1. Возвращать строго валидный JSON по схеме (без markdown/текста вокруг).
2. Никогда не предлагать intent, нарушающий `constraints`.
3. При низкой уверенности (`confidence < 0.35`) можно возвращать `intent_type = "none"`.
4. Любые недостающие поля должны заполняться безопасными дефолтами движка.

## Варианты провайдеров

### 1) Локальная модель (Ollama/vLLM)
**Описание:** модель разворачивается рядом с сервером симуляции (CPU/GPU). Запросы остаются в локальном контуре.

**Оценка:**
- **Стоимость:** низкая переменная стоимость (без per-token биллинга), но есть CapEx/OpEx на железо.
- **Задержка:** низкая/средняя, обычно стабильная в пределах локальной инфраструктуры.
- **Стабильность:** высокая при хорошем self-hosted SRE; зависимость от состояния локального узла.
- **Контекст:** ограничен выбранной моделью и VRAM/RAM; можно масштабировать через более крупные модели.
- **JSON-following:** от среднего до высокого при использовании грамматик/JSON schema constraints (лучше с guided decoding).

**Когда выбирать:** постоянные симуляции, чувствительные данные, предсказуемая нагрузка.

---

### 2) Бесплатный API с лимитами
**Описание:** внешний провайдер с free-tier, квотами и rate limits.

**Оценка:**
- **Стоимость:** минимальная на старте, но риск упереться в лимиты при росте нагрузки.
- **Задержка:** средняя/высокая, зависит от сети и общей нагрузки провайдера.
- **Стабильность:** средняя/низкая для production-SLA (возможны 429/5xx, дневные квоты).
- **Контекст:** часто ограничен бесплатным тарифом (меньше токенов/моделей).
- **JSON-following:** среднее качество; возможны отклонения без строгого response format.

**Когда выбирать:** ранний прототип, быстрый PoC без капитальных затрат.

---

### 3) Гибрид: локально по умолчанию, API для кризисов
**Описание:** основной путь — локальная модель; при кризисных событиях или деградации качества/latency — переключение на внешний API.

**Оценка:**
- **Стоимость:** умеренная; основная нагрузка дешево локально, API оплачивается только в «пиках важности».
- **Задержка:** низкая в обычном режиме, средняя в кризисном fallback на API.
- **Стабильность:** высокая при корректном failover и circuit breaker.
- **Контекст:** локально ограничен ресурсами, API покрывает «длинный контекст» для сложных кризисов.
- **JSON-following:** высокое при локальном constrained decoding + внешнем strict schema mode.

**Когда выбирать:** production-MVP с ограниченным бюджетом и требованием устойчивости.

## Профили переключения

### `mvp_low_end`
Цель: запуск на слабых машинах и минимальных затратах.

- По умолчанию: компактная локальная модель (или free-tier API при отсутствии локального рантайма).
- Ограничение контекста: короткий `memory_window`, агрессивная компрессия состояния.
- Частота вызовов: ниже (батчирование/кулдауны между AI-решениями).
- Политика качества: при любом сомнении — ранний fallback в deterministic policy matrix.

### `mvp_full_simulation`
Цель: максимальная глубина дипломатической симуляции.

- По умолчанию: гибридный режим (локальная модель + API для кризисов).
- Контекст: расширенный `memory_window`, больше сигналов `world_state`.
- Частота вызовов: выше, включая event-driven reevaluation.
- Политика качества: строгая валидация JSON, повторная генерация при schema violations, затем fallback.

## Graceful Degradation

Если AI недоступен, отвечает невалидно или превышены лимиты, система **обязана** перейти на deterministic policy matrix.

### Триггеры деградации
- Timeout выше порога (например, > 1500ms для low_end, > 2500ms для full_simulation).
- HTTP/network ошибки, 429, 5xx.
- Невалидный JSON или нарушение схемы после N ретраев.
- Circuit breaker открыт по метрикам ошибок.

### Поведение при деградации
1. Логировать причину (`provider_unavailable`, `invalid_json`, `rate_limited`, ...).
2. Применить deterministic policy matrix по тем же входам `context`.
3. Записать в telemetry источник решения: `decision_source = deterministic_fallback`.
4. Периодически делать health-check AI и возвращаться к AI only после успешных проб.

### Deterministic policy matrix (минимальные правила)
- Высокая мировая напряженность + плохие отношения -> `justify_war`/`none` (с учетом hard constraints).
- Низкая напряженность + нейтральные отношения -> `improve_relations`.
- Общий враг + достаточная совместимость -> `non_aggression_pact` или `faction_invite`.
- Недостаточно условий/конфликт ограничений -> `none`.

---

## Рекомендуемая стратегия для MVP
- Стартовать с `mvp_low_end` и локальной компактной моделью.
- Для стабильности сразу внедрить строгую JSON-схему + deterministic fallback.
- При росте сложности дипломатии перейти на `mvp_full_simulation` (гибридный профиль).
